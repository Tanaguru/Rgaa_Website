How to use the RGAA

The RGAA applies to any HTML content (HTML4, XHTML1, HTML5). For some tests, a reference baseline is used. This baseline takes into account a set of assistive technologies, browsers and operating systems, on which the accessibility of JavaScript-based interface components must be tested, among others. A detailed description is provided here: Baseline.

Important notice regarding HTML content prior to HTML5 specification

When the HTML code of the page is not HTML5, the HTML5 elements (tags and attributes) required by a criterion or test, are not applicable. Every other criteria or tests remain applicable, including those related to ARIA attributes, states or properties. The following criteria and tests are not applicable:

  • criterion 1.10;
  • criterion 9.2;
  • test 11.10.1 (condition 2, relative to the HTML5 attribute required).

Validation process

For each criterion, compliance is defined as follows:

  • Compliant (C): all applicable tests are passed
  • Non Compliant (NC): at least one applicable test is failed
  • Not Applicable (NA): there is no content targeted by the criterion.

The RGAA proposes two additional compliance statuses:

  • Derogated (D): some contents targeted by the criterion will meet compliance by derogation
  • Not tested (NT): the criterion has not been tested.

General guidelines

Give each image conveying information a relevant text alternative and a detailed description if necessary. Replace images of text with styled text when possible.

Key words

  • Principle: perceivable

Criterion 1.1 [A] Does each image conveying information have a text alternative?

  • Test 1.1.1: Does each image (<img> tag or element with the WAI-ARIA role="img" attribute) conveying information have a text alternative?
  • Test 1.1.2: Does each area (<area> tag) of an image map conveying information have a text alternative?
  • Test 1.1.3: Does each image button (<input> tag with the type="image" attribute) have a text alternative?
  • Test 1.1.4: Does each clickable area of a server-side image map have an equivalent link in the page?
  • Test 1.1.5: Does each decorative vector image (<svg> tag), conveying information, meet the following conditions?
    • The <svg> tag has a WAI-ARIA role="img" attribute.
    • The <svg> tag has a text alternative.
  • Test 1.1.6: Does each image object (<object> tag with the attribute type="image/…") conveying information, meet one of the following conditions?
    • The object tag has a text alternative.
    • The object tag is immediately followed by an adjacent link giving access to a page or a piece of text containing a relevant alternative.
    • A provided mechanism replaces the object by a relevant alternative.
  • Test 1.1.7: Does each embedded image (<embed> tag with the attribute type="image/…") conveying information, meet one of the following conditions?
    • The embed tag has a text alternative.
    • The embed tag is immediately followed by an adjacent link giving access to a page or a piece of text containing a relevant alternative.
    • A provided mechanism replaces the embed by a relevant alternative.
  • Test 1.1.8: Does each bitmap image (<canvas> tag) conveying information, meet one of the following conditions?
    • The <canvas> tag has a text alternative.
    • An alternative content is present between <canvas> and </canvas>.
    • The <canvas> tag is immediately followed by an adjacent link giving access to a page or a piece of text containing a relevant alternative.
    • A provided mechanism replaces the <canvas> element by a relevant alternative.

Criterion 1.2 [A] Is each decorative image properly ignored by assistive technologies?

  • Test 1.2.1: Does each decorative image (<img> tag), without caption, meet one of the following conditions?
    • The <img> tag has an empty alt attribute (alt="") and does not have any other attributes, that aim at labelling the image.
    • The <img> tag has a WAI-ARIA attribute aria-hidden="true" or role="presentation".
  • Test 1.2.2: Does each non clickable decorative area (<area> tag with no href attribute), meet one of the following conditions?
    • The area tag has an empty alt (alt="") and does not have any other attribute providing a text alternative.
    • The <img> tag has a WAI-ARIA attribute aria-hidden="true" or role="presentation.
  • Test 1.2.3: Does each image object (<object> tag with the type="image/…" attribute) conveying information, meet the following conditions?
    • The <object> tag has a WAI-ARIA aria-hidden="true" attribute.
    • The <object> tag doesn't have text alternative.
    • There is no text alternative between <object> and </object>.
  • Test 1.2.4: Does each decorative vector image (<svg> tag), not conveying any information, meet the following conditions?
    • The <svg> tag has an aria-hidden="true" attribute.
    • The <svg> tag and its children, don't have text alternative.
    • The <title> and <desc> tags are missing, or empty.
    • The <svg> tag and its children, don't have title attribute.
  • Test 1.2.5: For each decorative bitmap image (<canvas> tag), without caption, meet the following conditions?
    • The <canvas> tag has a WAI-ARIA aria-hidden="true" attribute.
    • The <canvas> tag and its children don't have text alternative.
    • The text content between <canvas> and </canvas> is empty.
  • Test 1.2.6: Does each decorative embedded image (<embed> tag with attribute type="image/…") without caption, meet the following conditions?
    • The <embed> tag has a WAI-ARIA aria-hidden="true" attribute.
    • The <embed> tag and its children don't have text alternative.

Technical notes

When an image is associated with a caption, the WCAG technical note recommends to systematically provide an alternative text for the image (see criterion 1.9). In this case the criterion 1.2 is not applicable.

In the case of a decorative vector image (<svg> tag) displayed through a <use href="..."> element, child from the <svg> element, the 1.2.4 test will also apply to the <svg> tag associated through the <use> tag.

A WAI-ARIA role="presentation" attribute may be used on decorative images and decorative non clickable areas. The "none" role introduced with ARIA 1.1 and synonym of the "presentation" role may also be used. However, it is still better to use the "presentation" role until the support for the "none" role is satisfying.

Technical notes

In the case of the SVG, the lack of support for the <title> and <desc> elements by assistive technologies creates a difficulty in the case of the implementation of an text alternative for the image or its detailed description. In this case, it is recommended to use the WAI-ARIA aria-label attribute to implement both a short text alternative and a reference to an adjacent detailed description or the WAI-ARIA aria-labelledby attribute to link the pieces of texts acting as short alternative and detailed description.

The use of WAI-ARIA aria-describedby attribute is not possible to link an image to its detailed description due to lack of support by assistive technologies.

The adjacent detailed description may be implemented using a <figcaption> tag, in this case, the 1.9 Criterion should be verified (notably with regards to the use of <figure> tags, WAI-ARIA role="figure" attribute and aria-label).

    • The bitmap image (<canvas> tag) and its caption are contained in a <figure> tag.
    • The <figure> tag has a WAI-ARIA role="figure" or role="group" attribute.

Technical notes

The implementation of a WAI-ARIA role="group" attribute or role="figure" on the parent <figure> element is meant to overcome the lack of current support <figure> elements by assistive technologies. The use of a <figcaption> element to link a caption to an image requires at least the use of a WAI-ARIA aria-label attribute on the <figure> parent element which content should be identical to the content of the <figcaption> element. To ensure optimal support, there can also be an explicit association between the content of the text alternative of the image and the content of <figcaption> element, for example:

<img src="image.png" alt="Photo : soleil couchant" /><figcaption>Photo : crédit xxx</figcaption>

The WAI-ARIA aria-labelledby and aria-describedby attributes cannot be used currently because of the lack of support by assistive technologies.

Note : additionally, captioned images should respect 1.1 et 1.3 Criteria related to images conveying information.

Technical notes

WAI-ARIA provides an aria-hidden attribute (true or false) that inhibits the rendering of a content to assistive technologies, without affecting its visibility for user agents: a content with aria-hidden="true" will not be spoken out but remains visible.

Unless the content controlled by aria-hidden is not intended to be rendered by assistive technologies, the value of the aria-hidden attribute must be consistent with the displayed or hidden status of the on-screen content.

The HTML5 specification describes a hidden attribute that can make unavailable (when the hidden attribute is present) content in the generated DOM (similar to type="hidden" on a form control).

There can be situations where a content controlled with hidden or aria-hidden is momentarily in a state inconsistent with the content’s displayed or hidden status, for example when an item is meant to be available but visually displayed only upon further action. In this case, it is the final state of the content that should be considered.
